# 具身智能综合服务平台 MVP 改造设计

日期：2026-05-14

## 1. 背景

原始需求文档希望基于 ShipAny 开发具身智能综合服务平台。当前实际可用底座是本仓库 `embodied-starter-vibe-to-production`，它已经提供 Next.js App Router、Better Auth、Drizzle/PostgreSQL、邮件订阅、后台权限、多语言、SEO 与基础后台框架。

本设计将需求中的 ShipAny 替换为当前模板，并聚焦第一阶段 MVP 主链路：

首页流量入口 -> 产品/企业检索 -> 用户提交需求 -> 后台处理线索

## 2. MVP 目标

本阶段只验证一个核心商业假设：

具身智能行业用户是否需要一个结构化信息平台，并愿意通过平台提交采购、合作或咨询需求。

成功标准：

- 用户能从首页快速理解平台定位。
- 用户能浏览和筛选产品、企业。
- 用户能从产品、企业或需求入口提交有效线索。
- 管理员能在后台查看、分配、跟进和关闭线索。
- 首批数据可以通过后台或脚本录入，不依赖完整 CMS。

## 3. 保留、隐藏与新增

### 保留

- 登录注册与管理员权限：复用 Better Auth 和现有 `admin` 角色。
- 数据库与 ORM：继续使用 PostgreSQL + Drizzle。
- 邮件能力：复用 Resend，用于线索通知和订阅。
- 国际化：继续使用 `next-intl`，MVP 以中文为主，英文可保留基础页面。
- SEO 基础：复用 metadata、sitemap、博客路由能力。
- 现有后台布局：在 `/admin` 下扩展业务管理模块。

### 暂时隐藏

- AI chat/image/video demo：不作为 MVP 主链路展示。
- 积分与订阅后台：保留代码，不接入本阶段业务。
- Creem 支付：保留，不进入 MVP 首发链路。

### 新增

- 产品目录与详情。
- 企业库与详情。
- 需求提交表单。
- 后台线索处理。
- 后台产品和企业基础管理。
- 搜索与筛选能力。

## 4. 信息架构

公开页面：

- `/`：首页
- `/products`：产品目录
- `/products/[slug]`：产品详情
- `/companies`：企业库
- `/companies/[slug]`：企业详情
- `/inquiry`：需求提交
- `/join`：厂商入驻申请
- `/blog`：行业资讯，第一阶段复用现有 MDX 博客

用户中心：

- `/dashboard`：保留现有用户中心入口
- `/dashboard/inquiries`：我的需求，P1，可在第一轮实现后补

管理后台：

- `/admin`：运营总览
- `/admin/products`：产品管理
- `/admin/companies`：企业管理
- `/admin/inquiries`：需求线索管理
- `/admin/vendor-applications`：厂商入驻审核

## 5. 核心用户流程

### 5.1 首次访客

1. 访问首页。
2. 看到平台定位、收录产品数、企业数、最新资讯和热门分类。
3. 进入产品目录或企业库。
4. 通过关键词、分类、行业、能力筛选目标对象。
5. 在详情页点击询价或联系。
6. 提交需求表单。
7. 系统保存线索，并通知管理员。

### 5.2 管理员

1. 登录管理员账号。
2. 进入 `/admin/inquiries`。
3. 查看新线索详情。
4. 设置处理状态：新线索、处理中、已联系、已关闭。
5. 记录内部备注和匹配建议。
6. 后续通过人工方式联系用户或厂商。

### 5.3 运营人员录入数据

1. 进入 `/admin/companies` 新建企业。
2. 进入 `/admin/products` 新建产品并关联企业。
3. 设置产品分类、行业、能力、价格区间和状态。
4. 前台列表和详情页自动展示。

## 7. 前台设计

### 7.1 首页

首页第一屏要表达三件事：

- 这是具身智能行业的一站式信息平台。
- 用户可以查产品、找企业、提交需求。
- 平台已有结构化数据积累。

首页模块：

- Hero：一句话定位、主搜索框、产品目录和提交需求按钮。
- 数据概览：企业数、产品数、资讯数、需求对接数。
- 快速入口：产品目录、企业库、行业资讯、需求对接。
- 热门分类：人形机器人、AGV、协作臂、四足机器人、机器视觉、导航定位。
- 最新产品和企业：展示少量最新发布内容。
- 邮件订阅：复用现有 newsletter 能力。

### 7.2 产品目录

列表页能力：

- 关键词搜索：名称、描述、厂商名称。
- 分类筛选：机器人、传感器、AI 系统、控制器、解决方案。
- 行业筛选：制造、医疗、物流、零售、农业、服务、教育。
- 能力筛选：搬运、识别、导航、抓取、交互、巡检。
- 价格区间筛选。
- 排序：最新、热度。

产品卡片展示：

- 封面图
- 产品名
- 厂商名
- 分类
- 关键标签
- 简短描述

### 7.3 产品详情

详情页展示：

- 产品名称、封面图、图片集。
- 厂商入口。
- 参数表。
- 应用场景。
- 产品描述。
- 官网链接。
- 询价/联系按钮。
- 同厂商或同分类相关产品。

询价按钮跳转 `/inquiry?sourceType=product&sourceId=...`。

### 7.4 企业库

列表页能力：

- 关键词搜索：企业名、简介、领域。
- 企业类型筛选：硬件厂商、AI 公司、系统集成商、服务商、研究机构。
- 地域筛选：国家、城市。
- 融资阶段筛选。

企业卡片展示：

- Logo
- 企业名称
- 一句话简介
- 所在地
- 企业类型
- 融资阶段

### 7.5 企业详情

详情页展示：

- 企业基础信息。
- 细分领域。
- 官网和社媒链接。
- 企业介绍。
- 旗下产品列表。
- 联系/合作按钮。

合作按钮跳转 `/inquiry?sourceType=company&sourceId=...`。

### 7.6 需求提交

表单字段：

- 姓名
- 公司名称
- 邮箱
- 手机
- 所属行业
- 需求场景
- 预算范围
- 详细描述

提交后：

- 写入 `inquiry`。
- 给管理员发送邮件通知。
- 展示提交成功页或成功状态。

## 8. 后台设计

### 8.1 产品管理

功能：

- 列表、搜索、筛选。
- 新建产品。
- 编辑产品。
- 发布、下架。
- 删除或归档。

第一阶段可以使用普通表单，不引入复杂富文本编辑器。

### 8.2 企业管理

功能：

- 列表、搜索、筛选。
- 新建企业。
- 编辑企业。
- 发布、下架。
- 删除或归档。

### 8.3 需求管理

功能：

- 查看需求列表。
- 按状态筛选。
- 查看需求详情。
- 修改状态。
- 记录管理员备注。
- 查看来源产品或企业。

第一阶段不做自动匹配，匹配由运营人工完成。

### 8.4 入驻审核

功能：

- 查看申请列表。
- 按状态筛选。
- 查看申请详情。
- 修改状态。
- 记录审核备注。

第一阶段审核通过后不自动创建厂商账号，只作为供给侧意愿验证。

## 9. 搜索方案

MVP 使用 PostgreSQL 基础查询实现：

- `ilike` 匹配名称、简介、描述。
- 多选筛选通过 JSON/text 数组字段匹配。
- 分页用 limit/offset。

暂不引入外部搜索服务。等数据超过数千条或需要更好的中文分词后，再评估 Meilisearch、Typesense 或 PostgreSQL full-text search。

## 10. SEO 与内容

- 产品详情和企业详情生成独立 metadata。
- 产品和企业详情加入 sitemap。
- URL 使用稳定 slug。
- 行业资讯第一阶段继续使用现有 MDX 博客。
- 若新增或删除博客文章，继续使用现有 `pnpm generate:blog-manifest` 流程。

## 11. 邮件通知

复用现有 `lib/email.ts` 和 Resend 配置。

触发邮件：

- 新需求提交：通知管理员。
- 新厂商入驻申请：通知管理员。

第一阶段不要求自动通知用户。提交成功页面即可确认。

## 12. 权限

公开可访问：

- 首页
- 产品目录和详情
- 企业库和详情
- 博客
- 需求提交
- 入驻申请

需要登录：

- 用户中心
- 后续的收藏和我的需求

需要管理员：

- 所有 `/admin/*` 业务管理页面

## 13. 非目标

本阶段不做：

- 厂商自助后台。
- 在线支付和会员权限。
- AI Agent 选型顾问。
- 复杂 CMS。
- 产品对比和测评体系。
- 自动线索匹配。
- API 开放。
- 复杂行为分析。

## 14. 实施阶段

### 阶段 1：数据模型和后台骨架

- 新增 Drizzle 表。
- 新增后台导航项。
- 新增产品、企业、需求、入驻申请基础管理页面。

### 阶段 2：前台检索闭环

- 首页改版。
- 产品目录和详情。
- 企业库和详情。
- 基础搜索筛选。

### 阶段 3：线索提交闭环

- 需求提交页。
- 产品/企业详情页接入询价入口。
- 后台线索状态处理。
- 管理员邮件通知。

### 阶段 4：数据与 SEO

- 准备首批企业和产品数据。
- 生成 slug。
- 补充详情页 metadata 和 sitemap。
- 运行 lint/build 验证。

## 15. 验收标准

- 首页能清楚表达平台定位，并能进入产品、企业和需求入口。
- 产品目录支持关键词、分类、行业筛选。
- 企业库支持关键词、类型、地区筛选。
- 产品详情能展示关联企业，并能发起需求。
- 企业详情能展示关联产品，并能发起需求。
- 用户可提交需求，数据库保存成功。
- 管理员可在后台查看线索并修改状态和备注。
- 新需求提交后管理员能收到邮件或在邮件未配置时不影响表单提交。
- 未登录用户可以提交需求。
- 非管理员无法访问后台业务页面。

## 16. 风险与处理

- 数据录入量大：第一阶段支持后台手动录入，后续可加 CSV 导入脚本。
- 中文搜索效果一般：先用基础关键词匹配，验证后再升级搜索引擎。
- 后台 CRUD 工作量偏大：第一版表单保持朴素，优先保证数据正确。
- 原模板 AI/支付功能干扰定位：先从导航和首页隐藏，不删除代码。
- 文章后台缺失：行业资讯先用 MDX 发布，避免第一阶段扩展 CMS。

## 17. 下一步

确认本设计后，进入实现计划拆分。建议第一轮实现只覆盖：

1. 数据表和基础数据访问函数。
2. 产品/企业列表与详情。
3. 需求提交表单。
4. 后台需求处理。

产品管理和企业管理可以先做最小 CRUD，复杂字段编辑和批量导入放到后续。
